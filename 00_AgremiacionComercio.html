🔷 Normalizaciones Aplicadas 3FN · Tercera Forma Normal
1 · Tabla Rol
Sin normalizar
Rol NVARCHAR(50)
'Administrador' | 'Auxiliar de Registro'
¿Por qué?
Evita texto repetido en cada fila de Usuario. Si el nombre del rol cambia, se actualiza un solo registro en el catálogo. Permite agregar nuevos roles sin alterar el esquema.
2 · Tabla Municipio
Sin normalizar
Municipio NVARCHAR(150)
'Bogotá' | 'bogota' | 'Bogotá D.C.'
¿Por qué?
Texto libre genera inconsistencias graves: variantes del mismo nombre se tratan como municipios distintos. El catálogo garantiza integridad referencial y confiabilidad en filtros y reportes.
3 · Tabla Estado
Sin normalizar
Estado NVARCHAR(20)
'Activo' | 'Inactivo'
¿Por qué?
Si el negocio requiere estados nuevos como 'Suspendido', solo se inserta un registro. Con texto plano habría que modificar el esquema y un CHECK constraint.
📌 Principio general — Las tres normalizaciones aplican 3FN (Tercera Forma Normal): eliminan dependencias transitivas y evitan que valores de dominio controlado vivan como texto libre en tablas de negocio. El resultado es un modelo más robusto, consistente y fácil de mantener a futuro.
⚡ Triggers de Auditoría AFTER INSERT · AFTER UPDATE
⚡ trg_Comerciante_Auditoria
Tabla
dbo.Comerciante
Evento
AFTER INSERT, UPDATE
¿Qué hace?
Cada vez que se inserta o modifica un comerciante, actualiza automáticamente FechaActualizacion con la fecha y hora exacta del momento, y UsuarioAuditoria con el usuario de la sesión SQL Server activa (SYSTEM_USER).
⚡ trg_Establecimiento_Auditoria
Tabla
dbo.Establecimiento
Evento
AFTER INSERT, UPDATE
¿Qué hace?
Cada vez que se inserta o modifica un establecimiento, actualiza automáticamente FechaActualizacion con la fecha y hora exacta del momento, y UsuarioAuditoria con el usuario de la sesión SQL Server activa (SYSTEM_USER).
🗂 Campos de Auditoría — Detalle
📅 FechaActualizacion
TIPO   DATETIME2(0)   NULL
Registra la fecha y hora exacta en que el registro fue creado o modificado por última vez. Es poblado automáticamente por el trigger usando GETDATE() — nunca se debe ingresar manualmente.
👤 UsuarioAuditoria
TIPO   NVARCHAR(255)   NULL
Registra el nombre del usuario de SQL Server que ejecutó la operación. Es poblado automáticamente por el trigger usando SYSTEM_USER. En producción puede reemplazarse por el usuario de la aplicación enviado desde el backend.

📊 Reporte de Comerciantes STORED PROCEDURE · FUNCTION
⚙️ sp_ReporteComerciantes
Tipo
STORED PROCEDURE
Ejecución
EXEC dbo.sp_ReporteComerciantes;
¿Qué hace?
Retorna todos los comerciantes con estado 'Activo', ordenados de forma descendente por cantidad de establecimientos. Por cada comerciante calcula automáticamente la cantidad de establecimientos, el total de ingresos y la cantidad de empleados usando la función auxiliar.
🔧 fn_ObtenerResumenEstablecimientos
Tipo
INLINE TABLE-VALUED FUNCTION
Retorna
TABLE (ComercianteId, CantidadEstablecimientos, TotalIngresos, CantidadEmpleados)
¿Por qué se creó?
Encapsula los cálculos de COUNT, SUM de ingresos y SUM de empleados agrupados por comerciante. Separarlo en una función independiente la hace reutilizable en otras consultas o procedimientos futuros, y mantiene el SP principal limpio y legible.

Detalles Técnicos
Filtro: Solo comerciantes Activos
Orden: DESC por cantidad de establecimientos
JOIN: LEFT JOIN con la función para incluir comerciantes sin establecimientos
Nulos: ISNULL muestra 0 o 'N/A' en lugar de vacíos
⚠️
Nota de Seguridad — Campo Usuario.Contrasena El campo Contrasena NVARCHAR(255) está definido para almacenar el hash de la contraseña, no el texto plano. El hashing debe realizarse en la capa de aplicación (backend) antes de persistir el dato, usando algoritmos como bcrypt, Argon2 o PBKDF2. SQL Server únicamente almacena el resultado. Nunca se debe guardar ni comparar contraseñas en texto plano.
